   245→نیازی به فراخوانی متد اعتبارسنجی تراکنش از سمت پذیرنده نیست.
   246→
   247→این حالت معادل فعال‌بودن اعتبارسنجی خودکار است.
   248→
   249→حتی اگر در پنل، اعتبارسنجی خودکار غیرفعال باشد، با ارسال این مقدار، تراکنش به‌صورت خودکار اعتبارسنجی خواهد شد.
   250→
   251→۲. اگر مقدار
   252→false
   253→ارسال شود
   254→در این حالت، اعتبارسنجی تراکنش به‌صورت خودکار انجام نمی‌شود و سیستم در هیچ مرحله‌ای بدون دخالت پذیرنده درخواست اعتبارسنجی به PSP ارسال نمی‌کند.
   255→
   256→زرین‌پال تنها زمانی اعتبارسنجی نهایی تراکنش را نزد PSP انجام می‌دهد که پذیرنده متد اعتبارسنجی تراکنش را فراخوانی کند.
   257→
   258→به‌عبارت دیگر، شروع فرآیند اعتبارسنجی نزد PSP وابسته به فراخوانی متد اعتبارسنجی تراکنش از طرف پذیرنده است؛ تا زمانی که این متد فراخوانی نشود، ارتباطی برای اعتبارسنجی با PSP برقرار نخواهد شد.
   259→
   260→اگر این فراخوانی در بازه‌ی زمانی مجاز انجام نشود، تراکنش نهایی نشده و مبلغ به حساب خریدار بازگردانده می‌شود.
   261→
   262→این حالت عملاً معادل انتخاب اعتبارسنجی غیرخودکار برای آن تراکنش است، حتی اگر در پنل درگاه، اعتبارسنجی خودکار فعال باشد.
   263→
   264→۳. اگر پارامتر
   265→auto_verify
   266→ارسال نشود
   267→در صورتی که پارامتر auto_verify در آبجکت metadata ارسال نشود، رفتار سیستم بر اساس تنظیمات درگاه در پنل زرین‌پال تعیین می‌شود:
   268→
   269→اگر در پنل، اعتبارسنجی خودکار فعال باشد، تراکنش به‌صورت خودکار اعتبارسنجی می‌شود.
   270→
   271→اگر اعتبارسنجی خودکار غیرفعال باشد، پذیرنده باید متد اعتبارسنجی تراکنش را فراخوانی کند تا فرآیند Verify نزد PSP انجام شود.
   272→
   273→نکات مهم
   274→استفاده از پارامتر auto_verify فقط در مواردی توصیه می‌شود که نیاز به کنترل لحظه‌ای رفتار اعتبارسنجی تراکنش وجود داشته باشد.
   275→
   276→در صورت عدم ارسال این پارامتر، رفتار سیستم کاملاً تابع تنظیمات درگاه شما در پنل زرین‌پال خواهد بود.
   277→
   278→در حالت اعتبارسنجی غیرخودکار، اگر متد اعتبارسنجی تراکنش از سمت پذیرنده در زمان مناسب فراخوانی نشود، پرداخت بازگشت خورده و عملاً از دست می‌رود.

   596→}

   599→شروع کار (Initialization)
   600→درخواست
   601→برای شروع روند احراز هویت، ابتدا باید یک درخواست POST با فرمت application/json به آدرس زیر ارسال کنید:

   604→لیست پارامترهای قابل‌پذیرش توسط این مقصد عبارت‌اند از:

   607→username        string  بله             آدرس ایمیل یا شماره‌ی همراه کاربر
   608→channel string('ussd', 'sms')   خیر     'ussd'  کانال دریافت رمز یک‌بار مصرف کاربر

   617→  -d '{"username":"09123456789","channel":"sms"}'

   622→  "avatar": "https:\/\/gravatar.com\/avatar\/{EMAIL_HASH}",
   623→  "channel": "ussd",
   624→  "ussd_code": "*733*4*97*2#",

   629→
   630→نام     نوع     الزامی است؟     توضیحات

   632→channel string('ussd', 'sms')   بله     کانالی که از طریق آن رمز یک‌بار مصرف در اختیار کاربر قرار می‌گیرد

   639→درخواست
   640→مشابه مرحله‌ی قبل، یک درخواست POST با فرمت application/json به آدرس زیر ارسال کنید:

   642→https://next.zarinpal.com/api/oauth/token
   643→لیست پارامترهای قابل‌پذیرش توسط این مقصد برای دریافت توکن (بدون Refresh Token) عبارت‌اند از:
   644→
   645→نام     نوع قابل‌قبول   الزامی است؟     مقدار پیش‌فرض   توضیحات
   646→grant_type      string  بله     'password'      نوع ورودی کاربر، در این مرحله باید برابر با 'password' باشد
   647→client_id       int     بله             آی‌دی کلاینت شما که از زرین‌پال دریافت کرده‌اید
   648→client_secret   string  بله             کلید کلاینت شما که از زرین‌پال دریافت کرده‌اید
   649→username        string  بله             آدرس ایمیل یا شماره‌ی موبایل کاربر
   650→password        string  بله             رمز یک‌بار مصرف که کاربر وارد کرده است
   651→scope   string  بله     '*'     @todo
   652→برای مثال، در ادامه‌ی مثال قبل، کاربری با شماره‌ی موبایل 09123456789، رمزِ 12345678 را وارد کرده است:
   653→
   654→$ curl 'https://next.zarinpal.com/api/oauth/token' \
   655→  -H 'Content-Type: application/json' \
   656→  --data-binary '{
   657→    "grant_type": "password",
   658→    "client_id": 100,
   659→    "client_secret": "$ecreT",
   660→    "username": "09123456789",
   661→    "password": "12345678",
   662→    "scope": "*"
   663→  }'
   664→پاسخ
   665→پاسخ این در درخواست (در صورت صحیح‌بودن داده‌ی ارسالی، بخش خطاها را ببینید) به صورت زیر خواهد بود:
   666→
   667→{
   668→  "token_type": "Bearer",
   669→  "expires_in": 1296000,
   670→  "access_token": "{ACCESS_TOKEN}",
   671→  "refresh_token": "{REFRESH_TOKEN}"
   672→}
   673→لیست جزئیات پاسخ‌های ارسال‌شده به این صورت هستند:
   674→
   675→نام     نوع     الزامی است؟     توضیحات
   676→token_type      string  بله     نوع توکن ارسالی رو مشخص می‌کند؛ نوع توکن‌های زرین‌پال Bearer هستند
   677→expires_in      int     بله     طول عمر توکن را به ثانیه مشخص می‌کند
   678→access_token    string  بله     Access Token برای کارکردن با امکانات سرور GraphQL؛ این توکن با استاندارد JWT (opens new window)ایجاد می‌شود
   679→refresh_token   string  بله     ‍Refresh Token برای دریافت توکن جدید بعد از منقضی‌شدن توکن قبلی؛ بخش Refresh Token را ببینید
   680→Refresh Token
   681→بعد از این که Access Token دریافتی پس از مدتی که در فیلد expires_in مشخص شده منقضی شد، لازم است که با استفاده از refresh_token داده‌شده در مرحله‌ی قبل، یک Access Token جدید دریافت کنید.

   683→درخواست
   684→برای دریافت توکن جدید، مشابه مرحله‌ی قبلی، یک درخواست POST با فرمت application/json به آدرس زیر ارسال کنید:
   685→
   686→https://next.zarinpal.com/api/oauth/token
   687→لیست پارامترهای قابل‌پذیرش به منظور دریافت توکن جدید با Refresh Token برای این مقصد به این صورت است:
   688→
   689→نام     نوع قابل‌قبول   الزامی است؟     مقدار پیش‌فرض   توضیحات
   690→grant_type      string  بله     'refresh_token' نوع ورودی کاربر، در این مرحله باید برابر با 'refresh_token' باشد
   691→client_id       int     بله             آی‌دی کلاینت شما که از زرین‌پال دریافت کرده‌اید
   692→client_secret   string  بله             کلید کلاینت شما که از زرین‌پال دریافت کرده‌اید
   693→refresh_token   string  بله             مقدار Refresh Token که از مرحله‌ی قبل دریافت کردید
   694→scope   string  بله     ''      @todo

   698→خطاها
   699→در صورتی که در هر یک از مراحل بالا، خطایی رخ دهد یا داده‌ی ارسال‌شده در درخواست، نامعتبر باشند؛ فیلد errors در پاسخ دریافتی حاوی یک آرایه خواهد بود که خطاهای درخواست در آن مشخص می‌شوند و فیلد data نیز مقدار null خواهد داشت.
   700→
   701→در این صفحه
   702→ثبت راهنمای استفاده از API
   703→به جز مراحل احراز هویت، بقیه‌ی امکانات API از طریق کارگزاری در‌دسترس هستند که با زبان پرس‌و‌جوی GraphQL کار می‌کند. برای پرس‌و‌جو (مثلاً گرفتن لیست تیکت‌ها) یا ایجاد تغییر روی داده‌ها (مثلاً ایجاد یک تیکت)، شما نیاز دارید یک عبارت به عنوان Query به سرور زرین‌پال با URL زیر ارسال کنید:
   704→
   705→https://next.zarinpal.com/api/v4/graphql
   706→این درخواست با استفاده از متد POST و با نوع محتوای application/json ارسال می‌شود. کد زیر، نمونه‌ی ارسال یک کوئری با ابزار cURL نشان می‌دهد:
   707→
   708→$ curl 'https://next.zarinpal.com/api/v4/graphql/' \
   709→  -H 'Accept: application/json' \
   710→  -H 'Authorization: Bearer {ACCESS_TOKEN}' \
   711→  --data-binary '{"query":"query { Application { application, platform } }","variables":null}'
   712→در نمونه‌ی بالا، با توکن {ACCESS_TOKEN} کوئری query { Application { application, platform } } اجرا می‌شود.
   713→
   714→در ادامه، خلاصه‌ای از چگونگی نوشتن پرس‌و‌جو با GraphQL آورده شده. در بخش شِمای مستندات، می‌توانید لیستی از پرس‌و‌جوها و تغییرهایی که می‌توانید ایجاد کنید را پیدا کنید.
   715→
   716→نوشتن پرس‌و‌جوی GraphQL
   717→پرس‌وجوی شما، می‌تواند از نوع query یا mutation باشد.
   718→
   719→query - اطلاعاتی را می‌خواند و تغییری ایجاد نمی‌کند
   720→
   721→mutation - اطلاعاتی را تغییر می‌دهد و نتایج تغییرات را برمی‌گرداند
   722→
   723→در ابتدا، نوع پرس‌و‌جوی خودتان را مشخص می‌کنید:
   724→
   725→query {
   726→
   727→}
   728→یا برای تغییر:
   729→
   730→mutation {
   731→
   732→}
   733→سپس باید نام عمل پرس‌و‌جو یا تغییر را تعیین کنید، برای مثال می‌خواهیم لیست کوئری‌های کاربر را بگیریم:
   734→
   735→query {
   736→  Tickets {
   737→
   738→  }
   739→}
   740→این کوئری هنوز صحیح نیست، چرا که سرور نیاز دارد که بداند شما به چه فیلدهایی لازم دارید. مثلاً ما فیلدهای id و status را لازم داریم:
   741→
   742→query {
   743→  Tickets {
   744→    id
   745→    status
   746→  }
   747→}
   748→نکته: برای نوشتن پرس‌و‌جوها در یک خط، بین فیلدها به جای n\ باید , قرار دهید، مثلاً:
   749→
   750→query { Tickets { id, status } }
   751→با ارسال این پرس‌و‌جو به سرور، پاسخی مشابه پاسخ زیر دریافت می‌کنید:
   752→
   753→{
   754→  "data": {
   755→    "Tickets": [
   756→      {
   757→        "id": "139805250316",
   758→        "status": "CLOSED"
   759→      },
   760→      {
   761→        "id": "139805250335",
   762→        "status": "CLOSED"
   763→      }
   764→    ]
   765→  }
   766→}
   767→پاس‌دادن پارامتر
   768→برخی از پرس‌و‌جوها و هم‌چنین همه‌ی تغییرها (Mutation) نیاز به پارامترهای ورودی دارند. مثلاً در مثال بالا، شما می‌توانید تعیین کنید که حداکثر ۱۵ تیکت به شما برگردانده شود:
   769→
   770→query {
   771→  Tickets(limit: 15) {
   772→    id
   773→    status
   774→  }
   775→}
   776→لیست پارامترها، فیلدهای بازگشتی و نوع آن‌ها را در شِما می‌توانید ببینید.
   777→
   778→و یا می‌توانید تعیین کنید که از ۱۰ تیکت اول صرف‌نظر شود:
   779→
   780→query {
   781→  Tickets(limit: 15, offset: 10) {
   782→    id
   783→    status
   784→  }
   785→}
   786→اما بیایید ایجاد یک تغییر را بررسی کنیم:
   787→
   788→mutation {
   789→  CardAdd(pan: "1111222233334444", expired_at: "2020-02-05 00:00:00") {
   790→    id
   791→  }
   792→}
   793→اولین چیزی که باید به آن دقت کرد، تفاوت نوع ریشه‌ی پرس‌و‌جو (Root Type) است. در مثال‌های قبلی، چون نیازی به نوشتن داده‌ی جدیدی نداشتیم، از query استفاده می‌کردیم، اما در این مثال، می‌خواهیم یک کارت جدید اضافه کنیم، پس به mutation نیاز داریم.
   794→
   795→در واقع انواع پرس‌و‌جوهایی که در دسترس شما هستند، یا از نوع query هستند یا mutation؛ برای استفاده از هر کدام باید Root Type عبارت پرس‌و‌جوی خودتان را مطابق آن قرار دهید.
   796→
   797→در پرس‌و‌جوی بالا، در صورت موفقیت‌آمیز بودن عملیات، id کارت جدید برگردانده می‌شود. در واقع Return Type عملیات CardAdd از نوع Card است، بنابراین باید تعیین کنید که دقیقاً کدام فیلدهای شیٔ Card ساخته‌شده را لازم دارید. در قسمت بعدی درمورد شیٔ‌ها توضیح داده شده است.
   798→
   799→نوع‌ها در GraphQL
   800→به طور کلی، GraphQL سه نوع داده وجود دارد:
   801→
   802→Object
   803→Scalar
   804→Enum
   805→نوع Scalar همان داده‌های معمول هستند که یکی از String، Int، DateTime، ID یا Boolean هستند. نوع Object یا شیٔ مانند بیشتر زبان‌های برنامه‌نویسی، از تعدادی عضو تشکیل شده که عضوهای آن را فیلد (Field) صدا می‌زنیم. هر فیلد از یک Key و یک Value تشکیل شده. Key حتماً یک رشته است که در میان بقیه‌ی فیلدهای دیگر یگانه است و Value هم از نوع Scalar یا Object یا لیست است. در GraphQL، نوع شیٔ با کلیدواژه‌ی type در شِما تعریف می‌شود، مثلاً:
   806→
   807→type Post {
   808→  id: ID!
   809→  title: String!
   810→  content: String
   811→  comments: [Comment!]
   812→}
   813→علامت ! در جلوی نوع فیلد به این معنی است که مقدار فیلد هیچ‌گاه Null نخواهد شد.
   814→
   815→هم‌چنین GraphQL از نوع دیگری به نام List پشتیبانی می‌کند، یعنی پاسخ پرس‌و‌جو یا تغییر شما، می‌تواند یک لیست از چندین شیٔ یا مقدار اسکالار باشد، مثلاً اگر شِما به این شکل باشد:
   816→
   817→query {
   818→  Tickets: [Ticket!]!
   819→}
   820→کد بالا نشان می‌دهد که عملیاتی به نام Tickets، مجموعه‌ای از اشیای Ticket برمی‌گرداند. علامت ! جلوی Ticket، به این معنی است که هیچ‌کدام از اعضای این لیست، Null نیستند. (در حال حاضر تمام لیست‌های GrpahQL باید به این گونه باشند و مقدار Null برای اعضا پشتیبانی نمی‌شود.) علامت ! جلوی براکت، به این معنی است که این لیست «حتماً» حداقل یک عضو دارد. اگر این علامت نباشد، به این معنی است که پاسخ عملیات Tickets می‌تواند یک لیست خالی ([]) باشد.
   821→
   822→مقادیر اسکالار هم می‌توانند به صورت لیست ارسال شوند، مثلاً اگر شِما به این شکل باشد:
   823→
   824→query {
   825→  Numbers: [Int!]
   826→}
   827→یک لیست که هر عضو آن یک Int است در پاسخ آن ارسال می‌شود.
   828→
   829→دقت کنید که داخل هر شیٔ هم می‌تواند یک فیلد با نوع لیست هم باشد.
   830→
   831→برای اطلاعات بیشتر درمورد نوشتن شِما و پرس‌و‌جوهای GraphQL می‌توانید به مستندات آن (opens new window)مراجعه کنید.
